You are building a new separate application inside the Red Hat demo/platform environment.

Application name:
DeepField

Purpose:
DeepField is a fleet-scale OpenShift signal intelligence and inference benchmarking platform.

It proves a repeatable Red Hat + Intel platform pattern:

Deterministic filters compress massive multi-cluster telemetry into a small number of high-value inference tasks, then measure how much fleet one Intel Xeon/Gaudi inference cluster can monitor.

The killer proof:
One inference cluster can monitor N OpenShift clusters because only X% of raw signals require expensive reasoning.

DeepField must also include a synthetic client/load generator that can benchmark the available inference cluster across models, routes, and workload profiles.

This is NOT StarGate.
This is NOT Launchpad.
This is NOT ARE.
This is NOT a governance demo.
This is NOT an autonomous remediation system.

DeepField is the signal intelligence and benchmark layer:
- generate synthetic fleet telemetry
- collect live demo platform cluster telemetry read-only
- normalize signals
- filter cheaply
- deduplicate
- correlate
- route only high-value findings to inference
- benchmark model endpoints under staged load
- measure compression ratio
- measure inference capacity
- estimate how much fleet one inference cluster can monitor

Primary principle:
Filter cheap. Reason expensive.

Secondary principle:
Benchmark with repeatability, not vibes.

Current available hardware context:
- Access to a large Intel Xeon 6 / Gaudi 3 cluster
- Approximately 2000 Xeon 6 CPUs
- Approximately 10.5 TB memory
- Available model endpoints include:
  - DeepSeek-R1-Distill-Qwen-14B on Gaudi 3, 4 HPUs
  - Microsoft Phi-4 on Gaudi 3, 2 HPUs
  - Qwen3-14B with 2 replicas, each on Gaudi 3, 4 HPUs
  - Llama 3.1 70B Q4 GGUF on Xeon 6767P CPU
- There may be live Red Hat demo platform clusters available for read-only monitoring.
- The platform should support both synthetic fleet testing and live demo platform cluster monitoring.
- The synthetic client should also support benchmark runs against the actual inference endpoints.

Build this in stages.
Use TDD.
Use red/green gates.
Do not build the whole universe in one pass.

PRODUCT SUMMARY
------------------------------------------------------------

DeepField is a fleet signal intelligence system for OpenShift and a repeatable benchmarking harness for the Intel/Red Hat inference cluster.

It answers:

1. How many clusters can one inference cluster monitor?
2. How many raw signals can we process per minute?
3. How much telemetry can deterministic filters discard or compress?
4. What percentage of signals require LLM reasoning?
5. Which model lane is best for which reasoning task?
6. What are the latency, throughput, and cost characteristics?
7. What is the reasoning compression ratio?
8. Where does the system saturate first?
9. How much inference capacity exists across the available Xeon/Gaudi endpoints?
10. Which model endpoint handles which workload best?
11. How does throughput change as concurrency ramps?
12. What is the max stable load before latency or error rate degrades?

The product must support three modes:

1. Synthetic Fleet Mode
   Generates simulated OpenShift cluster telemetry at controlled scale.

2. Live Fleet Mode
   Reads from real demo platform OpenShift clusters using read-only access.

3. Inference Benchmark Mode
   Uses the synthetic client to generate repeatable reasoning, summary, RCA, and fleet-analysis workloads against the inference cluster.

The MVP must work in Synthetic Fleet Mode and Mock Inference Mode first.
Benchmark mode should support mock endpoints first, then real endpoint adapters.
Live Fleet Mode should be adapter-based and read-only.

CORE CONCEPTS
------------------------------------------------------------

Raw Signal:
A low-level event or state observation from a cluster.

Examples:
- pod phase
- pod restart
- event warning
- route health
- PVC pending
- node pressure
- namespace quota
- VM state
- model endpoint health
- Kafka lag
- KServe endpoint readiness
- Launchpad session status, later

Normalized Signal:
A common representation of raw signals across clusters.

Candidate Finding:
A correlated group of signals that may be worth reasoning over.

Reasoning Task:
An expensive inference request generated from a candidate finding.

Final Insight:
A summarized finding that humans can act on or review.

Synthetic Benchmark Request:
A repeatable artificial inference request generated to benchmark model endpoints.

Benchmark Run:
A staged load test across one or more inference endpoints.

Inference Capacity:
The measured throughput/latency/error envelope of a model endpoint or model route.

Fleet Capacity Projection:
An estimate of how many clusters/namespaces/signals one inference cluster can monitor under defined latency and throughput targets.

------------------------------------------------------------
KEY PRODUCT KPIs
------------------------------------------------------------

Add these first-class KPIs:

1. Reasoning Compression Ratio

Definition:
raw_signals_received / reasoning_tasks_created

Example:
1,000,000 raw signals / 500 reasoning tasks = 2,000:1

2. LLM Escalation Rate

Definition:
reasoning_tasks_created / raw_signals_received

Example:
500 / 1,000,000 = 0.05%

3. Stable Model Throughput

Definition:
Highest concurrency level where:
- p95 latency remains under configured target
- error rate remains below configured target
- timeout rate remains below configured target

4. Projected Fleet Coverage

Definition:
max_raw_signals_supported_per_minute / average_raw_signals_per_cluster_per_minute

Where:
max_raw_signals_supported_per_minute =
max_reasoning_tasks_per_minute * reasoning_compression_ratio

5. Cost Per Finding

Definition:
estimated_reasoning_cost_units / final_insights_created

These should be shown prominently in reports and dashboard.

NANO-AGENT FILTER LAYER
------------------------------------------------------------

Implement deterministic nano-agents.

Nano-agents are tiny filters. They do not call LLMs.

Initial nano-agents:

1. PodHealthAgent
- detects crashloops, pending pods, image pull failures

2. RouteHealthAgent
- detects routes with no endpoints or unhealthy status

3. PVCHealthAgent
- detects PVC pending or stuck

4. NodePressureAgent
- detects memory/cpu/disk pressure signals

5. NamespaceQuotaAgent
- detects quota exceeded or near limit

6. KServeEndpointAgent
- detects model endpoint not ready

7. KafkaLagAgent
- detects high lag, if signals exist

8. LaunchpadSessionAgent
- detects active/failed/expired lab sessions, if signals exist

9. TransientSuppressorAgent
- suppresses known short-lived startup/provisioning transients

10. DedupeAgent
- deduplicates identical signals within a configurable time window

Each nano-agent must:
- accept NormalizedSignal list or stream
- return FilterDecision list
- be deterministic
- be fast
- be unit tested

TDD RED/GREEN MATRIX
------------------------------------------------------------

Build with TDD.

| Build Stage | Red Condition | Green Condition | Evidence |
|---|---|---|---|
| Domain Models | invalid signals accepted | valid/invalid models tested | unit tests |
| Synthetic Generator | nonrepeatable synthetic data | seeded generation deterministic | unit tests |
| Benchmark Client Models | invalid benchmark requests accepted | benchmark models validated | unit tests |
| Benchmark Generator | benchmark data not repeatable | seeded benchmark workload generated | unit tests |
| Normalizer | raw signals not normalized | normalized schema produced | unit tests |
| Nano Agents | filters missing or nondeterministic | deterministic filter decisions | unit tests |
| Dedupe/Suppression | duplicates pass through | duplicates/transients reduced | unit tests |
| Correlation Engine | no findings created | related signals become findings | unit tests |
| Signal Routing | everything goes to LLM | only eligible findings create tasks | unit tests |
| Mock Inference | no model metrics | deterministic inference metrics generated | unit tests |
